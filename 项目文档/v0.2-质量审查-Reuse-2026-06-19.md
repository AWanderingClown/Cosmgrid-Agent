## 1. 关键发现（5 条）

### 🔴 K-1：前端再发一次 sync 端点拿 token usage —— 浪费调用且可能与流结果不一致

**位置**：`app/src/pages/ChatPage.tsx:106-128`

流式响应已经完成，前端为拿到 `usage` 又用相同 `messages` 调一次 `/api/chat/sync`。这违反 v0.2 端到端一致原则：
- 两次调用 LLM（流 + sync），**账单 × 2**
- 第二次可能返回不同结果（`temperature` 非确定）
- 多 200-2000ms 延迟

**推荐**：在 `/api/chat/stream` 的 SSE 流末尾追加一段 `data: {...usage...}` 块，useChat 的 `onFinish` 直接读 `message.metadata.usage`（Vercel AI SDK 6 原生支持）。

**重要性**：P0 — 影响用户使用成本和延迟

### 🔴 K-2：ApiResponse 信封类型在前端 3 处重复定义 + 没有统一解析 helper

**位置**：
- `app/src/components/providers/AddProviderDialog.tsx:31-38`（`ApiOk<T> / ApiFail`）
- `app/src/pages/ChatPage.tsx:34-47`（`CredentialsResponse / ModelsResponse / StreamResponse`）
- `app/src/pages/ProvidersPage.tsx:48-58`（裸 `.then(r => r.json())` 然后 `if (b.success)`）

后端 `server/response.ts:6-15` 已定义 `ApiResponse<T>` 信封，前端却**重新发明**：
- 3 个文件、3 套不同的泛型
- `if (!body.success) throw new Error(body.error)` 在 AddProviderDialog 出现 3 次
- 类型不一致：AddProviderDialog 用 `ApiOk<T> | ApiFail` 联合类型，ChatPage 用 `success: boolean` 平铺

**推荐**：在 `app/src/lib/api.ts` 集中：
- 导出 `ApiResponse<T> = ApiOk<T> | ApiFail`（直接用后端类型）
- 导出 `apiFetch<T>(url, init): Promise<T>` —— 内部 fetch + json + 解包 + 错误抛出

**重要性**：P0 — 这是后面所有 API 调用的基础设施，越晚改越痛

### 🟡 K-3：后端 zod schemas 已有 14 张表的 `z.infer` 类型，前端却全部手写 interface

**位置**：

**问题**：前端手写类型 ≠ 后端 schema 字段。当前已经出现 drift 风险：
- 后端 `provider.type` 是 `string`（v0.2 简化未约束枚举）→ 前端 `ProviderTypeValue` 用 `as const` 收紧

**重要性**：P1 — 推迟到 v0.3 会更痛（4 个新页面将再写 4 套类型）

### 🟡 K-4：AddProviderDialog 串 3 个 API 调用 —— 应该有 batch 端点

**位置**：`app/src/components/providers/AddProviderDialog.tsx:82-137`

当前流程：
1. `POST /api/providers` → 拿 providerId
2. `POST /api/api-credentials` → 拿 credentialId
3. `POST /api/models` → 拿 modelId
4. `PATCH /api/api-credentials/{id}` 回填 defaultModelId

4 个 HTTP 调用 = 4 次 401 鉴权、4 次 CORS 握手、4 次 Zod 验证、4 次 Prisma 写入。

**推荐**：后端加 `POST /api/providers/bundle` 端点：
```ts
// server/routes/bundle.ts
bundleRouter.post("/", zValidator("json", createBundleSchema), async (c) => {
  const tx = await prisma.$transaction([
    prisma.provider.create({ data: input.provider }),
    prisma.apiCredential.create({ data: { ...input.credential, providerId: ... } }),
    prisma.model.create({ data: { ...input.model, providerId: ... } }),
  ]);
  // 回填 defaultModelId
  await prisma.apiCredential.update({
    where: { id: tx[1].id },
    data: { defaultModelId: tx[2].id },
  });
  return ok(c, { provider: tx[0], credential: tx[1], model: tx[2] });
});
```

**优点**：
- 4 次调用 → 1 次（前端代码量减半）
- 单事务保证原子性（v0.2 当前如果第 2 步失败，孤儿 provider 留在 DB）
- 服务端校验"3 个 entity 互引用"的一致性

**重要性**：P1 — 业务逻辑正确性问题（partial failure 留垃圾数据）

### 🟢 K-5：3 处 fetch + 状态变更可抽成 `useApiResource` 自定义 hook

**位置**：
- `app/src/pages/ProvidersPage.tsx:42-96`（`providers / credentials / models` 三态手动管理 + `load()` 函数 + 4 个删除 handler + 1 个 toggle handler）
- `app/src/pages/ChatPage.tsx:50-78`（`models / loadError` 状态 + IIFE 加载）

ProvidersPage 的 `load()` 模式：
- 3 个 `useState`
- 1 个 `Promise.all` 并行拉
- 1 个 `load()` 重新拉
- 4 个写操作后**硬调** `load()` 刷新

**推荐**：用 v0.3 计划引入的 `react-query` / `swr`：
```ts
// 简单替代（不引新依赖）—— 自写 useApi hook
function useApiResource<T>(url: string) {
  const [data, setData] = useState<T[]>([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const reload = useCallback(async () => {
    setLoading(true);
    try {
      const res = await apiFetch<{ data: T[] }>(url);
      setData(res.data);
    } catch (e) { setError(getErrorMessage(e)); }
    finally { setLoading(false); }
  }, [url]);

  useEffect(() => { void reload(); }, [reload]);
  return { data, loading, error, reload, setData };
}
```

**重要性**：P2 — v0.2 还能扛，v0.3 加项目工作区后状态机复杂度 ×3

```ts
// ============ ApiResponse 类型（与后端 server/response.ts 对齐）============
export interface ApiOk<T> {
  success: true;
  data: T;
  meta?: { total: number; page: number; limit: number };
}
export interface ApiFail {
  success: false;
  error: string;
}
export type ApiResponse<T> = ApiOk<T> | ApiFail;

// ============ apiFetch: 统一 fetch + json + 信封解包 + 错误抛出 ============
export class ApiError extends Error {
  constructor(message: string, public readonly status?: number) {
    super(message);
    this.name = "ApiError";
  }
}

export async function apiFetch<T>(
  url: string,
  init?: RequestInit,
): Promise<T> {
  const res = await fetch(url, {
    ...init,
    headers: {
      "Content-Type": "application/json",
      ...init?.headers,
    },
  });
  const body = (await res.json()) as ApiResponse<T>;
  if (!body.success) {
    throw new ApiError(body.error, res.status);
  }
  return body.data;
}

// ============ 集中类型（v0.2 先手写，v0.3 切 z.infer）============
export interface Provider {
  id: string;
  name: string;
  type: string;
  website: string | null;
  notes?: string | null;
}
export interface ApiCredential {
  id: string;
  name: string;
  baseUrl: string;
  enabled: boolean;
  providerId: string;
  provider: { name: string; type: string };
  defaultModelId: string | null;
}
export interface Model {
  id: string;
  name: string;
  displayName: string | null;
  contextWindow: number | null;
  enabled: boolean;
  workRoles: string; // JSON 字符串（与后端一致）
  providerId: string;
  provider: { name: string };
}

// ============ API 资源端点常量（避免裸字符串散落）============
export const API = {
  providers: "/api/providers",
  credentials: "/api/api-credentials",

  modelsFilter: "/api/models-filter",
  chat: {
    stream: "/api/chat/stream",

} as const;
```

**复用方案**：单事务 + 1 次调用

新建 `app/server/routes/bundle.ts`：

```ts
// POST /api/providers/bundle —— 一次创建 Provider + Credential + Model
// 走单事务保证原子性（避免 partial failure 留孤儿数据）
import { Hono } from "hono";
import { z } from "zod";
import { prisma } from "../db.js";
import { ok, fail, handleZodError } from "../response.js";
import {
  createProviderSchema,
  createApiCredentialSchema,
  createModelSchema,
} from "../schemas.js";

export const bundleRoutes = new Hono();

const bundleInputSchema = z.object({
  provider: createProviderSchema,
  credential: createApiCredentialSchema.omit({ providerId: true, defaultModelId: true }),
  model: createModelSchema.omit({ providerId: true }),
});

bundleRoutes.post("/", async (c) => {
  try {
    const input = bundleInputSchema.parse(await c.req.json());

    // 事务保证 3 个 entity 一致性
    const result = await prisma.$transaction(async (tx) => {
      const provider = await tx.provider.create({ data: input.provider });
      const credential = await tx.apiCredential.create({
        data: { ...input.credential, providerId: provider.id, defaultModelId: null },
      });
      const model = await tx.model.create({
        data: { ...input.model, providerId: provider.id },
      });
      // 回填 defaultModelId
      const updatedCredential = await tx.apiCredential.update({
        where: { id: credential.id },
        data: { defaultModelId: model.id },
      });
      return { provider, credential: updatedCredential, model };
    });

    return ok(c, result);
  } catch (error) {
    if (error instanceof Error && error.name === "ZodError") {
      return handleZodError(c, error);
    }
    return fail(c, 500, "批量创建失败");
  }
});
```

`server/index.ts:178-181` 注册：
```ts

app.route("/api/providers/bundle", bundleRoutes);
```

`AddProviderDialog.tsx:82-141` 改写为：
