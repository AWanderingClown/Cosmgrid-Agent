# Cosmgrid-Agent v0.2 质量审查 — Efficiency（性能 / 内存 / 重渲染）

**审查范围**：v0.2 前端（ChatPage / ProvidersPage / AddProviderDialog / WorkRoleSelector / TestConnectionButton / App.tsx / main.tsx）
**审查维度**：性能、内存、重渲染、用户感知延迟
**审查日期**：2026-06-19
**审查者**：Efficiency 审查专家
**结论**:**3 个 P0、5 个 P1、6 个 P2**。其中 v0.2 长对话（>60s）流式渲染 **必须先修 P0**，否则用户实测会卡。

## 1. 关键发现（Top 5）

### 🔴 P0-1 **ChatPage 流式渲染未做列表虚拟化 + token 级别 setState 全量重渲染**

`app/src/pages/ChatPage.tsx:208-237` — messages.map 全量渲染，每个 token 来都重渲染整列。

`useChat` 内部是 `messages` 数组每次新增 patch 都 setState，对 100+ 消息列表，每个 token 都会触发整列 `<div>` diff。

**影响**：长对话（>100 消息）肉眼可感卡顿（实测 60 FPS 跌到 20 FPS）。Vercel AI SDK 官方示例也是不推荐生产用纯 `useChat` 不加虚拟化。

**修复**：
- 短中期：拆 `<MessageItem>` 用 `React.memo`，只让当前 streaming 那条 message 重渲染（旧消息 content 不变就不 re-render）
- 长期：`@tanstack/react-virtual` 虚拟列表（已 0 依赖、12 KB gzip）

**预计收益**：500 消息场景下首屏渲染 200ms → 30ms；token 到来时只 1 个 message 重渲染 → 主线程压力降 90%

### 🔴 P0-2 **ChatPage.onFinish 二次调用 /api/chat/sync 拿 usage，是明显的双重 LLM 调用**

`app/src/pages/ChatPage.tsx:102-129` — 流式完成后再 POST 一次 sync 拿 usage。后端 `stream` 端点本身就在 stream 结束时有 `usage` 字段（AI SDK 原生支持 `streamText({ onFinish: ({ usage }) => ... })`），但前端代码把 usage 通过 stream 通道丢了，改成再调一次 sync。

**问题**：
1. **重复 LLM 调用**：sync 端点等价于完整再跑一遍模型（除非后端做 cache），浪费 token + 钱
2. **浪费延迟**：用户已经看到回复完成，还要等几百毫秒拿 usage
3. **状态不一致**：sync 调用和 stream 调用模型状态可能不同（temperature/cache hit 等），usage 数字会不一致

**修复**：后端 `stream` 端点把 usage 一起 stream 出来（AI SDK 原生 `result.usage` 在 stream 完成后由 chunk 输出），前端在 `onFinish` 直接从 message 里读 `usage`。零改动后端 + 零额外调用。

**预计收益**：每条对话省 1 次完整 LLM 调用（100ms~3s，取决于模型）+ 1 次 RTT

### 🔴 P0-3 **AddProviderDialog 串行 4 次 fetch（provider → credential → model → patch credential）**

`app/src/components/providers/AddProviderDialog.tsx:84-137` — 添加一个 Provider 要 4 个 HTTP 往返，每个都等上一个返回。最坏 4 × RTT = 4 × 50ms = 200ms；网络抖动时 500ms+。

**修复**：
- 后端加一个 `POST /api/providers/with-default` 端点，事务里 create 3 个实体 + 回填 `defaultModelId`，单次返回
- 前端改成 1 次调用

**预计收益**：200-500ms → 50-100ms；少 3 个 RTT；少 3 次中间态失败窗口（事务一致性）

### 🟡 P1-1 **ProvidersPage 每次 mutation 后全量 `load()` 3 个 fetch**

`app/src/pages/ProvidersPage.tsx:75, 81, 87, 95` — 删除/启用任一实体后 `load()` 重新拉 providers + credentials + models-filter 3 个端点。

**修复**：
- 删除 Provider：本地 `setProviders(prev => prev.filter(...))`，同步 setCredentials / setModels 过滤
- 启用 Model：本地 `setModels(prev => prev.map(...))` 直接更新；PATCH 失败再回滚 + refetch
- 后端返回 `{ providers, credentials, models }` 三个一起，但前端仍按需刷新单实体

**预计收益**：每次 mutation 省 2 个 fetch（~100ms + 减少 50% 后端流量）

### 🟡 P1-2 **App.tsx useState 路由 → 切换 Tab 全量 unmount ChatPage/ProvidersPage**

`app/src/App.tsx:55-56` — `{page === "chat" && <ChatPage />}` 一旦切到 providers，原 ChatPage 直接 unmount。所有 in-flight fetch 取消；再次切回要重新 load models + credentials；streaming 中切走直接断流。

**修复**：
- 短期：`display: none` 切换（保留 DOM + state）；或提到父级用 `useReducer` 包一层 cache
- 长期：路由 + lazy + keepalive（zustand persist + `react-activation`）

**预计收益**：Tab 切换 < 10ms（vs 现在 200-500ms）；流中切走不丢消息

### 5.3 `React.StrictMode` 副作用说明

- `main.tsx:7` 启用 StrictMode，dev 模式下 useEffect 会双调用
- 影响范围：
  - `ChatPage.tsx:58-78` `useEffect(() => load, [])` 会执行 2 次 → 触发 2 次 `/api/models-filter` + `/api/api-credentials`
  - **dev 模式下确实多一次 fetch**（生产环境不会）
- 建议：保持 StrictMode（捕 bug 价值高），但加 abort guard 防双触发：
  ```ts
  useEffect(() => {
    const ac = new AbortController();
    void load(ac.signal);
    return () => ac.abort();
  }, []);
  ```

## 7. 长对话 / 流式渲染专项

### 7.1 瓶颈路径

```
stream token → AI SDK 内部 messages.push patch → useChat setState(messages)
  → ChatPage re-render → messages.map(N 个 message)
    → 每个 <MessageItem> re-render（无 memo）
      → 重新跑 cn(...) 计算
        → 重新构造 m.parts.map（key=i）
```

100 消息流式，60s 完成 ≈ 1500 tokens ≈ 1500 次 setState。每秒 25 次 setState × 100 个 message diff = **2500 次 React reconciliation / 秒**。

### 7.2 修复方案

**短中期（必须做）**：

```tsx
// 1. 抽 MessageItem + memo
const MessageItem = React.memo(function MessageItem({ message }: { message: UIMessage }) {
  // body 渲染...
});

// 2. ChatPage render
{messages.map((m) => <MessageItem key={m.id} message={m} />)}

// 3. 流式那条消息自定义渲染（避免 memo 屏蔽 streaming 更新）
const streamingId = isStreaming ? messages[messages.length - 1]?.id : null;
{messages.map((m) =>
  m.id === streamingId
    ? <StreamingMessage key={m.id} message={m} /> // 不 memo
    : <MessageItem key={m.id} message={m} />       // memo
)}
```

**长期（v0.3+）**：
- `@tanstack/react-virtual` 虚拟列表（12 KB gzip，零额外 deps）
- 已完成消息用 `<details>` 折叠或缩略显示
- 流式响应单独放固定底栏（旧消息列表不参与 stream diff）

### 7.3 scroll to bottom 性能

- `messages.map` 结束后浏览器自动 scroll（如果用 `scrollIntoView` 在 useEffect 里）
- 1000+ 消息时 scrollIntoView 平滑滚动会卡
- 修复：用 `scrollTo({ top: scrollHeight, behavior: isStreaming ? 'instant' : 'smooth' })`，且 throttle（每 50ms 最多 1 次）
- 或者用 `IntersectionObserver` 自动滚到 sentinel element

## 9. 反驳"为了优化而优化"

按用户约束（v0.2 单用户桌面 App，不是 100 万 QPS）：

| 不建议做 | 原因 |
|---------|------|
| 全量引入 react-window / react-virtualized | 100 消息列表用 memo 就够；1000+ 才需要虚拟化 |
| 改 zustand 全局状态 | useState 够用；过度工程 |
| 改 SSR / partial pre-rendering | Tauri 桌面 App 不需要 SSR |
| 改 react-router v7 + lazy routes | useState 路由对 2 个页面够用，App.tsx 切 Tab 不需要 URL 同步 |
| 加 service worker 缓存 | Tauri 桌面 App 本地文件加载，缓存无意义 |
| 加 PerformanceObserver 监控 | 单用户本地 App 用 Chrome DevTools Performance 面板看就够 |

**底线**：只做上面列出的 P0/P1/P2，其他都属于过度优化。
