## 1. 改进（修真半成品 —— 这些是已写但不真能用的）

### 改进-1 🔴 SmartRouter 冷启动死锁 → 出厂先验 + 隐式信号（**最高优先，痛点3 的命门**）

**病根**：`model-performance-stats.ts:13` 阈值=30，`isScoreEligible` 要 30 条同 taskType 样本才走评分路径，但唯一调用方是它自己的测试 → 真实路由层从不用 score → **评分路径是死代码**。新用户/新项目永远走 v1，永远凑不够 30 条，**永远学不到** = 冷启动死锁。这恰好是用户最在意的"真省钱"环节。

**解法（两段）**：
1. **出厂贝叶斯先验**：每个模型已有 `capabilityScore`（出厂能力分）。把它当先验均值，样本作为后验更新（简单做法：`score = (prior·k + Σsample) / (k + n)`，k≈5 当伪计数）。**1 条样本就能微调，不必等 30**。门槛从"30 才生效"改成"始终生效、置信度随样本上升"。
2. **隐式信号采集**（用户点出的方向，现状 UsageEvent **零 outcome 字段**，已核实）。给 UsageEvent 加 `outcome` 字段，采集：
   - **失败信号**：同一 message 上 regenerate（上个模型不行）；用户手动从便宜模型切回贵模型（路由派错）；写操作被拒绝确认弹窗 / edit 后立刻又 edit 同文件 / git snapshot 被 revert（AI 输出质量差）。
   - **成功信号**：确认弹窗点了批准且之后没回滚；对话自然结束没重试；对弈里 judge 选中了某模型方案。
   - 这些 outcome 作为 reward 喂回 performance-stats，把"靠 token 数/延迟"的弱信号升级成"靠真实采纳率"的强信号。

**产出**：SmartRouter 从纸面变真实，痛点3 打分 2 → 3+。这是本方案投入产出比最高的一项。

### 改进-2 🟡 embedding 占位 → 语义缓存对中文真生效

**病根**：`embedding.ts:68` `keywordEmbeddingProvider` 是关键词哈希，CJK 只切到二元组，中文同义改写基本击不中 → 语义缓存退化成"精确文本匹配"，省不了 token。

**解法**：spike `transformers.js` 跑 all-MiniLM-L6-v2（或更小的多语模型）在 Tauri WebView 的兼容性（MEMORY 已记为 TODO）。先 2h spike 验证能不能在 WebView 跑 onnx；不行再退到"jieba 中文分词 + 改善哈希"的保底方案。
**注意**：这条依赖改进-1 才有意义 —— 缓存省下的钱要能在 StatsPage 上被"成功信号"统计出来，否则又是 0。

### 改进-3 🟡 cli-engine abort 不 kill 子进程（痛点1 相关，白烧订阅额度）

**病根**：`cli-engine.ts:112` TODO 原文"abort 后 CLI 仍在后台跑完（白耗额度）"。用户限额时切走，旧的 spawn claude 还在跑、还在扣订阅额度 —— 跟"省钱"直接冲突。

**解法**：abort 时通过 Rust 侧 `spawn_cli_stream` 持有的 child handle 真 `kill()`。需在 lib.rs 加一个 `kill_cli(session_id)` command + TS 侧 abort 时调用。小改动、高价值。

## 2. 增强（补能力 —— 补 OpenCode 硬伤 + 闭环缺口）

### 增强-1 🔴 LSP 桥（getDiagnostics）—— 比 MCP 更该先做

**为什么第一**：现在工具层只有"读/改文件+跑命令"，**AI 改完代码看不到自己写错了**（全目录搜 LSP/getDiagnostics 零命中，已核实）。对 vibe coder 这是致命的——他看不懂代码，全指望"AI 自测自报结论"，但 AI 自己都不知道编译挂了。这是 agentic 闭环的关键缺口，也是 OpenCode 的真实硬伤。

**解法**：工具层加一个 `getDiagnostics(filePath)` 工具，桥接 `typescript-language-server`（或先用最轻的：edit 后自动跑 `tsc --noEmit` 抓该文件错误）。AI 改完→自动看诊断→自己修→再确认。把"改文件"升级成"改文件并验证"。

### 增强-2 🟡 git 读类工具（status/diff/log）—— 对弈和工具循环都要用

**病根**：`git-snapshot.ts` 只写不读，无 git_status/git_diff/git_log（已核实）。AI 做多步工具循环"改完→看 diff 确认效果"这步根本没数据可看。

**解法**：复用现有不经 sh -c 的 git 调用模式（lib.rs git_commit_file 那套独立 argv），加 git_diff / git_status 两个只读工具。让 AI 能"看见自己改了什么"。

### 增强-3 🟡 复杂问题自动建议对弈（差最后一根线）

**病根**：`classifyMessageComplexity` 已存在且能判 hard（已核实），但**没联动** —— ChatPage 遇到架构/分析类问题不会提示"要不要开对弈"。痛点2 卡在"用户根本没入口想起来用对弈"。

**解法**：ChatPage 在 classify 返回 hard 且含对比/选型/架构类词时，气泡式提示"这个问题适合多模型对弈，点这里 3 个模型一起上"。一根线的事，直接把痛点2 的功能从"藏起来"变"主动浮现"。

### 增强-4 🟢 MCP 接入口子（可排在 LSP/git 之后）

竞品报告把 MCP 列为硬伤，但**用户 4 个痛点里没有 MCP**。它是"工具能力可扩展"的长期价值，不是当下命门。建议：留一个 stdio MCP 接入口子（跟现有工具层并列），但排在 LSP/git/对弈联动之后。

## 3. 明确不做 / 暂缓（守边界，省力气）

| 项 | 决策 | 理由 |
|---|---|---|
| bash 改 sidecar | **暂不做** | `sh -c` + 危险模式/白名单/项目黑名单三道闸 + `$()`反引号黑名单已够安全；sidecar 是洁癖不是刚需 |
| git worktree 隔离 | **暂不做** | 单文件 commit 对 vibe coder 够用；worktree 是程序员多分支需求，不在用户边界内 |
| 技能/插件市场 | **永不做** | OpenClaw ClawdHub 20% 恶意技能的活教训；坚持工具内置 |
| 多频道桥接（TG/邮件/X） | **永不做** | OpenClaw 致命三要素翻车现场；只接本地项目目录 |
| web_search/web_fetch 工具 | **缓** | 不在痛点内，优先级低于上述全部 |
| debate 5b 评测框架 | **缓** | 等用户真跑几次对弈收集真实反馈，再决定要不要 LLM-as-judge |

## 5. 用户必须本人做的一步（我做不了）

**端到端冒烟测试** —— Tauri GUI + 真实 API key，CLI 环境点不了、也不该动你订阅额度。建议你本人花 10 分钟跑一遍，或上 Playwright（已在 deps，e2e-runner 可用）自动化：

1. ChatPage 问"读 package.json" → 看 read 工具被调
2. 让 AI 改一个文件 → 弹确认框 → 点批准 → 文件真改了 → git commit 落库 → ToolExecution 落库
3. 配一个故意 401/429 的主模型 + 一个有 key 的备用 → 问一句 → 看顶部"已无缝切到 XXX"+ 对话历史完整带过去
4. 开一次对弈 → 看 solver/critic/judge 三面板 + UsageEvent 落 debate_* + 最终方案合理

跑完这一步，痛点1/2 的打分才能从"demo 级"升到"真实用过"。这也是 StatsPage 从全 0 到有数据的唯一途径。

## 6. 还没落进本方案、待你拍板的"让上下文可感知"（待办②）

用户待办②"让'上下文是资产'对用户可感知"是产品哲学层，不是 bug，单独拎出来给你选方向（建议 LSP 之后做）：

- **A. 上下文护照条**：ChatPage 顶部常驻一条，显示当前上下文带了什么（项目记忆 N 条 / 历史 M 轮 / 检查点 K 个）+ "已服务过的模型"头像串（claude→glm→kimi）。每切一次模型，头像串加一个 + "上下文 0 丢失"绿标。让用户**肉眼看见"工人换了、资产没动"**。
- **B. 限额切换 toast 升级**：从"已切到 XXX"升级成"额度用完，已无缝换 XXX 接手，上下文 100% 带过去（N 条记忆 + M 轮历史）"。
- **C. 检查点/接力包做成游戏存档式可视化**，强化"这是我的资产"。

这三个不冲突，可叠加。建议至少先做 A（最直接、改一个组件）。
