## 0. 总览（TL;DR）

v0.2 前端**功能跑通了，但有 3 个治标设计、3 个浅层 TODO、2 个死代码/未连接点**，必须在 v0.2 内修 1 个（明文存 API Key）、加 TODO 注释 5 个、删死代码 1 个（crypto.ts）。

**关键判断**：
- **crypto.ts 是死代码**（0 处引用）——v0.2 直接删，留 1 行 commit message 解释。
- **API Key 明文存 DB**（AddProviderDialog.tsx:105）——v0.2 **必须**改，因为 Tauri 桌面应用 DB 文件就在用户磁盘上，明文就是裸奔。
- **流式响应的 usage"打补丁取法"**（ChatPage.tsx:106-128）——v0.2 不治本（后端 SSE 协议改动跨前后端），但必须留 TODO，v0.3 改 SSE 末尾 metadata 块。

**3 个治本方案 v0.2 必做**：
1. API Key 至少前端不存 DB——v0.2 改后端 `apiKeyEncrypted` 字段从明文改为「占位符 `***`，调用时从前端 session 拿」（或最简单：**v0.2 后端临时只接受明文，前端每次用都从 UI 重输，不持久化**）。
2. 删 `lib/crypto.ts`（死代码 + 反向教学，告诉新人"我们用了安全加密"，但实际没接）。
3. `AddProviderDialog` 三段串联：v0.2 在前端加"成功后才算成功，失败要清理已建资源"——但 v0.2 时间紧的话留 TODO + 注释。

## 1. 关键发现（Top 5 治标问题）

### 🔴 P0-1：API Key 明文存 SQLite（AddProviderDialog.tsx:105）

```typescript
// AddProviderDialog.tsx 第 105 行
apiKeyEncrypted: apiKey, // v0.2 临时：明文存 DB（v0.3 改 OS keychain）
```

**问题**：
- Cosmgrid-Agent 是 Tauri 桌面 App，DB 文件路径在用户机器上：`~/Library/Application Support/...`（macOS）/ `%APPDATA%`（Windows）。
- 任何能读用户 home 目录的进程（备份工具、磁盘清理、家人/同事、电脑送修）都能直接拿到所有 Provider 的 API Key。
- 注释写"v0.3 改 OS keychain"——但 v0.2 上线后用户真实用过，DB 里就已经有真实 Key 了。
- 用户原话「不要托底临时处理」——这就是典型的"现在能用、底层是雷"。

**严重性**：🔴 P0（生产事故风险，且不治本永远爆）

**治本方向**（二选一，v0.2 必做其一）：
- **方案 A（推荐，半天搞定）**：v0.2 后端把 `apiKeyEncrypted` 字段改成"不持久化"，调用时强制前端每次传明文。代价：每次开 App 要重输 Key（用户体验差，但**安全**）。
- **方案 B（推荐，1 天搞定）**：v0.2 接 `tauri-plugin-stronghold` 或 `tauri-plugin-keyring`，API Key 只存 OS keychain，DB 里只存 `keychainRef: "anthropic-prod-001"`。这是 v0.3 原计划的提前。

**v0.2 内能否做**：✅ 能做（方案 A 改 1 个后端字段 + 1 个前端调用路径；方案 B 加 1 个 tauri 插件）。

### 🔴 P0-3：ChatPage 流式 usage 取法是"双倍烧钱 + 用户看到假数据"

```typescript
// ChatPage.tsx 第 102-129 行
onFinish: ({ message }) => {
  if (message.role === "assistant") {
    // 简化：让 sync 端点也跑一次拿 usage（流式不返回 usage 给前端）
    void fetch("/api/chat/sync", {
      method: "POST",
      ...
      body: JSON.stringify({
        modelId: selectedModelId,
        credentialId: selectedCredentialId,
        apiKey,
        messages: messages.map((m) =>...),  // ← 把刚发完的完整对话再发一遍
      }),
    })
```

**问题**：
- 流式响应结束时，前端**再调一次 `/api/chat/sync` 把完整对话重发一遍**，只为拿 usage。
- 后果 1：**用户每次对话双倍扣费**（流式 1 次 + sync 1 次 tokens 全算）。
- 后果 2：sync 端点和 stream 端点响应可能不一致（温度、max_tokens、model 版本），用户看到的 usage 可能不准。
- 后果 3：长对话时 `messages.map(...)` 全量重发，**第二次调用 token 消耗 = 第一次 2 倍**（prompt cache 没命中）。

**根本原因**：后端 SSE 流协议没在响应末尾加 usage metadata 块。AI SDK v5 的 stream protocol 天然支持 `onFinish` 回调里返回 `usage`，但**后端响应体里没塞**。

**严重性**：🔴 P0（每个用户每次对话都被扣双倍费用）

**治本方向**：
- v0.2 跨前后端：后端 stream 末尾加 `data: {"type":"finish","usage":{...}}` 块，前端 `useChat` 直接消费（AI SDK v5 原生支持）。
- 时间不够 v0.2 改后端的话：**v0.2 前端至少在 sync 调用前 `console.warn("⚠️ v0.2 双倍计费已知问题，v0.3 修")`**，让用户知情。

**v0.2 内能否做**：✅ 后端改 SSE 协议 + 前端移除 sync 调用。跨端但工作量 1 天。

### 🟡 P1-4：AddProviderDialog 三段串行 fetch 无回滚

```typescript
// AddProviderDialog.tsx 第 84-141 行
// 1. POST /api/providers           ← 建了
// 2. POST /api/api-credentials     ← 建了
// 3. POST /api/models              ← 失败！
// → 数据库里留了 provider + credential 孤儿记录
```

**问题**：
- 任何一步失败（model 名重复、credential name 冲突、网络断），前两步创建的记录**已经 commit 到 DB**，下次打开 App 能看到"半截 provider"。
- 用户体验：以为"添加失败"，刷新发现表里多了个没模型的 provider，**得手动清理**。

**根本原因**：前端在串 3 个独立的 REST API，**每个 API 不知道前面/后面还有步骤**，所以也不会回滚。

**严重性**：🟡 P1（用户能看到，能手动恢复，但不修的话每次添加失败都留垃圾）

**治本方向**（v0.2 必做其一）：
- **方案 A（推荐，后端 1 天）**：新建 `POST /api/setup-provider` batch endpoint，后端用 Prisma `$transaction([...])` 包三个 create，任一失败自动 rollback。前端只调 1 次。
- **方案 B（前端 1 小时）**：v0.2 加"失败回滚"逻辑——try/catch 里手动 `DELETE /api/providers/{id}`、`DELETE /api/api-credentials/{id}`。脆弱（中途网络断就漏），但比现在好。

**v0.2 内能否做**：✅ 方案 A 1 天，方案 B 1 小时。

## 7. 审查方法论备注

本次审查用的 3 个问题：
1. **修复深度**——这段代码修复到根因了，还是只盖住表面症状？
2. **治标 vs 治本**——如果只治标，临时方案撑到哪个版本会爆？
3. **一般化 vs 特殊处理**——这段代码是通用模式（适用于未来场景）还是硬编码特殊场景？

**示例**：TestConnectionButton 的 2s debounce = 治标（防客户端点击）+ 治本缺失（防服务端被刷）。当前只治标，**根因是后端没限流**——v0.2 必须补后端。

**反例**：crypto.ts = 治本假象（"我们加密了"）+ 实际未接 = 死代码 + 误导 = 双重债，必须删。

**反面教材**：ChatPage sync 取 usage = 治标（"先让用户看到数字"）+ 副作用（双倍计费）= 用户不知道自己在付双倍 = P0。

## 8. 总结

v0.2 前端**功能跑通**，但有 1 个 P0 安全雷（明文存 Key）+ 1 个 P0 财务雷（双倍计费）+ 2 个 P0 架构雷（事务缺失 + 死代码）+ 3 个 P1 TODO 缺失。

**v0.2 必须修 5 项 P0 + 加 6 处 TODO + 删 1 个死代码**，工作量约 3-4 天。

**不修会怎样**：
- API Key 明文 → 用户 Key 泄露 → Anthropic 账单爆炸 → 用户骂
- 双倍计费 → 用户余额减半 → 用户骂
- 三段无事务 → 用户看到孤儿数据 → 困惑
- 死代码 → 新人误读代码 → 错误决策

**治本 vs 治本假象**：crypto.ts 是"治本假象"（看起来做了，实际没接），删了反而干净；ChatPage 双倍计费是"治标"（先给用户看数字，但没治根因），v0.2 必须治本。

---

**审查结论**：v0.2 上线前必须完成 P0 五项（删 crypto.ts / API Key 不存明文 / 流式 usage 治本 / 测试连接后端限流 / batch endpoint）。P1 加 TODO，P2 写 backlog。**不要带着"v0.3 再改"的注释上线**——因为用户原话「以后都扯蛋」，每个 TODO 必须今天就明确写清，否则就是埋雷。
